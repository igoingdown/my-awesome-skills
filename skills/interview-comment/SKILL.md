# Interview Comment（面试评价生成）

按团队固定模板生成后端/算法/大数据研发岗位的面试评价。Claude 亲自读简历截图（视觉）+ 语音转写，产出双重评价报告：

1. **候选人评价**：五维打标 + 综合评分 + 长处/短处速判
2. **面试官复盘**：五维评估 + 做得好/不好 + 改进建议 + 时间分配分析

**只评估本轮材料，不读取、不引用、不被其他面试官的评价或结论影响。**

## 触发

用户说类似：

- **直接贴一个飞书招聘候选人链接**（`https://<tenant>.feishu.cn/hire/talent/<talent_id>?application_id=<application_id>...`）——走「阶段 0：材料采集」全流程，采集完成后自动进入评估
- "评估 `<人名目录>` 的第 N 轮"
- "帮我写 `<人名>/002` 的面试评价"
- "运行 interview-comment，候选人在 `<人名目录>`，我做的是 2 面"

## 阶段 0：材料采集（用户只贴飞书招聘链接时执行）

用户只给一个飞书招聘链接时，先自动完成简历与文字记录采集，产出标准目录结构后再进入评估流程。材料目录根为 `~/github/interviews/`。

### 0.1 打开页面

优先用 Claude Code 的 Chrome 集成（`claude --chrome` 启动 / `/chrome` 连接，需 Claude in Chrome 扩展）操作浏览器；若不可用，回退 AppleScript 方案（需 Chrome 菜单开启 查看→开发者→"允许 Apple 事件中的 JavaScript"）：

```bash
open -a "Google Chrome" "<链接>"   # 等待 5-6 秒加载
osascript -e 'tell application "Google Chrome" to get {URL, title} of active tab of front window'
# 页面 title 即候选人姓名，如 "郜忠明 - 招聘"
```

姓名转拼音得到目录名（如 郜忠明→gaozhongming），`mkdir -p ~/github/interviews/<拼音>/00N`（本轮为第几面就用几，1 面=001）。

### 0.2 下载简历 PDF → 渲染 resume.png

飞书招聘的简历接口（浏览器内带 cookie fetch，不要在终端直接 curl）：

```
GET /atsx/api/application/get_default_resume/?talent_id=<talent_id>&application_id=<application_id>
→ data.default_attachment.url 为 PDF blob 下载地址（带签名 token）
```

用 AppleScript 在页面内执行 JS：fetch 该 API 拿 url → fetch url 取 blob → FileReader 转 base64 存 `window.__pdfB64`。然后分块导出（AppleScript 单次返回有长度限制，按 10 万字符/块切片拼接）：

```bash
for i in 0 1 2; do osascript -e "tell application \"Google Chrome\" to execute active tab of front window javascript \"window.__pdfB64.slice($i * 200000, ($i + 1) * 200000)\"" >> pdf_b64.txt; done
tr -d '\n' < pdf_b64.txt | base64 -d > resume.pdf && file resume.pdf   # 校验是 PDF
```

渲染高清长图（多页纵向拼接，3x 缩放）：

```bash
swift ~/.claude/skills/interview-comment/scripts/pdf2png.swift resume.pdf resume.png
```

用 `sips -Z 800` 生成缩略图并 Read 目检，确认内容完整清晰。

### 0.3 抽取面试文字记录 → asr.md

1. 在页面右侧找到「面试速记」入口并点击（元素：`[class*=minutes]` 且文本为"面试速记"）：
   ```js
   Array.from(document.querySelectorAll('[class*=minutes]')).find(e => e.textContent.trim() === '面试速记').click()
   ```
2. 速记面板出现在 `[class*=tertiaryContainer]` 容器，消息块结构：`[class*=section__]` 内含 `[class*=name__]`（说话人）、`[class*=subInfo__]`（说话人+时间）、`[class*=content__]`（内容）。
3. **先滚动到底确认全量**（`tc.scrollTop = tc.scrollHeight`，等 2-3 秒后比对 section 数是否变化——目前观察为全量渲染非懒加载，但必须验证），检查是否有多段录制/分段 tab。
4. 页面内 JS 遍历全部 section 组装 JSON 存 `window.__asrJson`，按 1 万字符/块分片导出到本地，Python 校验：条数、首末条、空内容数、说话人集合（应恰好为面试官+候选人两人）。
5. 生成 `asr.md`：头部写候选人/面试官/岗位/时间/条数元信息，正文按 `**说话人 HH:MM**` + 内容逐条排列，**不重不漏、保持原始顺序、不改写原文**。

### 0.4 采集完成检查

目录内应有：`resume.pdf`、`resume.png`、`asr.md` 三个文件。逐项确认后进入正式评估流程（下方「执行流程」）。清理中间文件（`pdf_b64.txt`、`asr_raw.json`）。

### 0.5 并发提示（Dynamic Workflow）

进入评估时，**并行**发起两个 general-purpose 子代理：①候选人五维证据提取（读 asr.md + resume.pdf，产出证据密集的结构化清单）；②面试官复盘证据分析（问题全列表/时间分配/追问精准度/纪律/表达效率）。主对话同时亲自 Read asr.md 与 resume（视觉），交叉校验子代理结论后裁决打分并写报告。子代理只做证据提取，**打分裁决必须由主对话完成**。

## 输入目录结构（固定）

顶层是候选人标识（姓名 / 缩写 / 工号），下面每一轮面试一个子目录，命名为 `001`、`002`、`003`、`004`，分别代表 1~4 面。

```
<人名>/
├── 001/
│   ├── resume.png        # 简历截图（可能存在）
│   └── asr.md            # 该轮面试语音转文字（用户本人担任面试官）
├── 002/
│   ├── resume.png
│   └── asr.md            # ← 假设用户本次做的是 2 面
└── ...
```

**文件命名规则（严格匹配，不要用模糊搜索）**：

| 文件 | 含义 | 出现在 |
|------|------|--------|
| `resume.png` | 简历截图 | 通常每轮都有（允许多轮共享同一张） |
| `asr.md` | 该轮面试的语音转文字 | **仅用户本人担任面试官的轮次** |

> 注：候选人目录下可能存在其他面试官写的评价文件（如 `review-res.md`）。**本 skill 一律忽略——不读取、不解析、不纳入评价。** 每轮评价只基于该轮 `asr.md` + 简历独立成文。

### 判定"本次要评估哪一轮"

- 只有含 `asr.md` 的轮次才是**本次评估的目标轮**（评价要写到那一轮的目录里）
- 如果多个轮次都有 `asr.md` → 用 `AskUserQuestion` 让用户指定要评估哪一轮
- 如果没有任何一轮有 `asr.md` → 报错退出（缺少必要材料）
- 用户已在对话里明确指定轮次（如"帮我写 002"）→ 以用户指定为准

### 必需文件与可选文件

- **必需**（目标轮必有）：`asr.md`
- **强烈建议**（目标轮应有）：`resume.png`。若目标轮没有，查找其他轮是否有 `resume.png`，有就共用；全都没有则报错退出。

缺少 `asr.md` 或所有轮都无 `resume.png` → 直接报错退出，不脑补。

## 评分体系（核心）

### 分数档位

候选人的综合评分**只能**取以下 6 个值之一：

```
2   2.5   3   3+   3.5   4
```

**禁止**输出 2.75、3.25、3.75 等非档位分数。

### 档位含义

| 分数 | 含义 |
|------|------|
| 4 | 顶尖候选人，各维度均有明显亮点，能立即独当一面 |
| 3.5 | 优秀，2+ 个维度有显著加分项 |
| 3+ | 合格偏上，有明确亮点（通过线） |
| 3 | 合格线边缘，默认倾向不通过；面试官可基于实际体验显式裁定通过（须写明理由） |
| 2.5 | 有明显硬伤，不通过 |
| 2 | 明显不符合岗位要求 |

### 3+ 的定义

**3+ 及以上**（包含 3+、3.5、4）为通过。

**通过标准**：

- **综合评分 ≥ 3+：通过。**
- **综合评分 = 3：合格线边缘。默认按严格原则倾向不通过；但面试官可基于实际面试体验显式裁定通过/不通过。** 裁定通过时，必须在概览写明"打 3，面试官裁定通过"及具体理由（如：核心维度有真实底子，某些短板属入职后可习得/可锻炼，不构成阻断）。
- **综合评分 ≤ 2.5：不通过。**

> 注：开"3 可裁定"这个口子是为了容纳"核心能力达标、个别短板可后天补"的真实场景，但**不改变严格原则的整体精神**——3 默认仍偏保守，不写明裁定理由就按不通过处理；3 与 3+ 之间犹豫时不要直接跳到 3+，而应落 3 再由面试官决定是否裁定通过。

### 严格打分原则（硬规则）

1. borderline 一律按**低档**打：
   - 在 **3 和 3+ 之间犹豫 → 打 3**（落 3 后默认倾向不通过，由面试官决定是否显式裁定通过；不要因犹豫直接跳到 3+）
   - 在 3+ 和 3.5 之间犹豫 → 打 3+
   - 在 3.5 和 4 之间犹豫 → 打 3.5
   - 任何"可打可不打高一档"的情况 → 打低一档
2. 宁可错杀，不可错放
3. 给 3+ 必须说得出**至少 2 条**证据确凿的加分理由
4. 给 ≥3.5 必须说得出**至少 1 条**显著亮点（加分项，非持平）
5. **资历 × 深度期望校准（硬规则）**：分数不是绝对的，要按候选人资历（工作年限 + 当前/目标职级）校准深度要求。**资历越高，同样的回答应换来越严的判定。**
   - **先确认目标岗位对标职级**：期望线优先按**目标岗位职级**建立（如岗位对标 2-1 就按 2-1 期望判，即使候选人年限更长）；目标职级不明确时再按候选人资历。评估前从上下文推断或直接问用户"本岗位对标什么职级"。
   - **先建立期望线**：8 年 / 2-2（或更高）的候选人，对其主导/自研项目的核心组件，预期是"能讲清关键设计的取舍依据 + 答得出核心原理"；对中等算法题预期"能落地一个正确解法"。把候选人实际表现**对照其资历应有的深度**来判，而不是对照应届生。
   - **达不到资历期望即降档**：高资历候选人若出现"关键参数靠拍/忘、自研方案核心组件原理答不出、中等算法题写不出"，**这是系统性深度缺失而非单点瑕疵**——对应维度判 `-`，综合按判分铁律压到 ≤2.5，**不得**因表达流畅、广度真实、背景光鲜而上浮。
   - **同一份回答、不同资历可给不同分**：应届生"自研 agent 参数拍的"可记 `=`（尚在学习期），8 年 2-2 同样表现记 `-`（这个年限早该想清楚为什么这么设）；反之按 2-1 校准时，基础概念盲区（如 CAP、一致性哈希）可判"可入职后补齐"，不必触发铁律。报告概览须写明"按 职级 X（或 N 年）的深度期望校准"。
   - **面试官现场观察可参与裁定**：ASR 转写体现不出的信号（反应速度、吸收提示的速度、冲劲、现场给过提示导致的失分）由面试官口头补充后，可作为裁定依据修正评分；修正后在概览留痕初判与上调/下调理由，供后续轮次参考。
6. **警惕"广度/流畅"冒充深度（反陷阱）**：候选人把一个项目讲得完整、流畅、有量级，**不等于**该项目经得起深度追问。判 `+`（尤其技术支撑/技术深度）前，必须确认其**核心组件/关键设计**在被追问时答得出原理与取舍——**核心组件原理答不出，整个项目的"亮点"要打折，不能用流畅讲述撑起 `+`**。典型反例：自研高性能服务讲得头头是道，但其核心数据结构（如无锁队列）候选人没主动提、被问原理答"不太清楚"——此时该方案不构成技术深度加分。

## 五维度定性判断

五个维度：**业务理解、技术支撑、技术广度、技术深度、软素质**

每个维度用 `+` / `=` / `-` 标注：

| 标注 | 含义 |
|------|------|
| `+` | 加分项：该维度有明显亮点，有证据支撑 |
| `=` | 持平：符合岗位预期，无明显亮点或硬伤 |
| `-` | 减分项：该维度有明显不足或答错 |
| 未考核 | 本轮未覆盖 → **留空、不记录**（不要勉强给 `=`） |

### 各维度含义

#### 1. 业务理解

候选人对所做业务的理解深度。

- 核心判断：能否讲清业务目标 → 技术方案的映射
- 是否用业务指标而非单纯技术指标衡量成败
- 是否关心 ROI、成本、SLA
- **后端**：能否讲清接口/系统服务的上游业务、量级、关键 SLO
- **算法**：能否讲清模型优化指标 vs 业务指标的差异、badcase 驱动迭代经验
- **大数据**：能否讲清数据产品的消费方、SLA、数据 → 决策链路

#### 2. 技术支撑

候选人的技术能力能否真正支撑业务目标。

- 核心判断：技术选型的决策链路是否清楚、trade-off 意识、上线后真实表现
- 是否能讲出"为什么不选另一种方案"
- 是否有量级估算习惯
- `-` 红旗：把"我们用 Redis"当成设计题答案；扩展性追问直接卡死

#### 3. 技术广度

候选人技术栈的覆盖面。

- 核心判断：岗位相邻领域是否接触过、能否把不同组件串起来
- **注意**：广度 ≠ 列一堆名词，能讲清每个东西的**适用场景**才算广度
- `-` 红旗：简历列一堆"熟悉 X/Y/Z"，但任意挑一个都说不出和同类的区别

#### 4. 技术深度

候选人在至少一个领域的深度。

- 核心判断：被追问原理时能答到第几层
- 简历上写"精通 / 熟悉 X"是否经得起追问
- 关键区分：是**背答案**还是**真懂**（追问时是否崩溃/绕开/换话题）
- **AI 时代的归属校验**：候选人陈述的技术点，可能是 AI（编程工具/助手）替他写或说的，而非他本人掌握。对其方案里的**关键技术点抽查原理**（如"你用了 Lua，它在这里解决什么？"→ 应答"保证多命令原子性"）。**答不出 → 该技术点不计入其掌握**；若是他**主动提交的方案**却讲不清核心原理，从重处理（接近"答错"而非"没答出"）。
- 区分"没答出"（知识盲区，可接受）vs"答错了"（原理讲反，严重扣分）
- `-` 红旗：把核心概念讲反；简历明确写"精通 X"被问 X 基础原理答不出

##### ⚠️ "概念型选手"识别（八股 vs 实战）——必判不行

高学历/大厂/海归背景 + 表达流畅书面化的候选人，最容易被高估。这类人看过很多"八股"、能复述概念，但缺乏对关键技术的准确判断与实战落地。**一旦命中以下任一信号，对应维度判 `-`，不要被背景和流畅度洗白：**

1. **堆概念、无技术细节**：问"X 怎么实现的"，只会复述定义/比喻（如"skill 就像训练新员工把过程告诉他""大概几百行吧"），始终落不到具体实现 → 技术深度 `-`。
2. **架构理由站不住**：问某个架构决策（如"为什么不用 multi-cluster / 分库分表 / 某方案"），给的理由经不起一句反问就崩（如"原系统本来就是大 key，跟你的改造有什么关系？"→ 候选人承认"没仔细考虑过"）→ 技术支撑 `-`。**列了多个候选方案 ≠ trade-off 加分**，要看理由是否成立。
3. **关键技术答反**：刁钻题问到核心机制，候选人自信答错（如"多个 Agent 能否共用一个 MCP Server？"——正解：**STDIO 是单进程 stdin/stdout 管道，不能多路复用，必须为每个 Agent 起独立 server 实例**；答"共用相同 credential 不会有问题"是把 I/O 复用问题误判成鉴权问题，**答反**）→ 技术深度红旗 `-`，从重。
4. **表达抓不住重点**：自我介绍/答题反复"漂移"，抓不住核心。**别把"信息量大 / 书面化流畅"误读成"结构化"**——结构化是结论先行、点出价值与难点，不是话多。

**判分铁律**：核心维度（技术深度 / 技术支撑）被追问穿或答反 → 即便其他维度有 `+`，综合 **≤ 2.5 不通过**；前面的 `+` 被实质抵消。诚实承认边界是软素质中性项，**不能**用来把技术硬缺口洗成加分。**判 `+` 前先自问：这个亮点扛住面试官的犀利追问了吗？**

**岗位细化关注点**：

- **后端**：并发与锁、MySQL 索引/事务/MVCC、缓存一致性、MQ 语义、分布式 CAP
- **算法**：模型原理（loss/优化器/反向传播）、训练/推理工程瓶颈、实验设计与指标体系
- **大数据**：Spark/Flink 执行原理、shuffle、数据倾斜、state/exactly-once、数据建模分层

#### 5. 软素质

沟通、学习、价值观、稳定性的综合判断。

- 回答是否结构化（先结论再展开）
- 对追问的反应（积极重组 vs 不耐烦/抵触）
- 跳槽节奏是否合理、对前司是否明显情绪化
- 对"近期学了什么"能否答出有说服力内容
- **工作抽象与自我表达**：能否主动、有重点地讲出自己工作的价值与难点（结论先行 / STAR）。**判定规则：若候选人靠面试官逐层追问才讲出真实工作内容与难点（自己开口讲不出重点、抽象不出价值），此情形不作为软素质减分，但明确不能作为加分项（软素质最高只能 `=`）。** 可交叉看简历是否同样"堆功能、不点价值"。
- **红线**：价值观问题（如对同事的不当评价、歧视言论、"卷同事"当荣誉）→ 直接 `-`，且综合分不得高于 2.5

## AI coding 题专项评估

当本轮编码/设计题是 **AI coding 题**（允许候选人使用任意 AI 编程工具、投屏，面试官扮演产品经理只答非技术问题）时，**评估的是"用 AI 的过程与落地细致度"，不是代码能否跑通**（代码 AI 随时能写）。

### ⚠️ 不能只凭 asr 转写文本判 AI coding 题

转写记录的是"候选人最终说出的话"，其中很可能包含**AI 产出的完整、书面化文本**——这会让候选人显得很强、流程很完整，但**那是 AI 的完整，不是候选人能力的完整**。真实水平藏在屏幕操作过程里（怎么写第一个 prompt、有没有选型、有没有测试、被问才补还是主动做）。**评估必须结合面试官现场观察 / 屏幕过程；遇到转写在编码环节突然变得高度书面化、结构化，更要警惕这是 AI 产出而非本人现场表达，不可因"转写流畅"就给高分。**

### 过程清单（必需项，逐条核对）

任一项缺失，在对应维度如实体现（多为 AI 应用能力、技术广度的待提升，必要时落到技术支撑/技术深度）：

1. **需求澄清**：是否主动挖隐含条件/边界（如是否多机、限流维度、量级、并发要求），还是直接把需求原样扔给 AI 取第一版。
2. **方案选型**：是否对比多方案再决策（如限流的令牌桶 / 漏桶 / 滑动窗口），还是照单全收 AI 给的单一方案、无 trade-off。
3. **过程拆解**：是否先让 AI 出设计 / 拆 Task / 定验证（Verification）Plan，再分块实现，而非一句"帮我写个 X"。
4. **质量保障**：写完是否做**功能测试 + 性能测试**，到可交付状态。
5. **代码 review**：过程中是否对 AI 产出做交叉 code review。
6. **原理归属校验**：对 AI 生成的关键技术点抽查原理（见「技术深度 · AI 时代的归属校验」）。讲不清 → 该技术点不计入其能力；主动提交的方案讲不清核心原理，从重。

### 判分锚点

- **加分（AI 应用是真能力）**：主动澄清隐含条件 + 做方案选型对比 + 先方案后代码 + 主动写功能/性能测试 + 对产出能讲清原理并 review。能"现场驱动 AI 高效逼近一个可交付方案"。
- **地板（不加分）**：把需求交给 AI 让它写出来、能复述"标准 AI 协作流程"，但选型/测试/校验都靠面试官追问才补。
- **减分**：直接扔题不选型、不做测试、被问才改关键设计（如多机）、AI 生成的关键技术点讲不清原理（归属校验失败）。

> 提示：本节与团队既有的"识破不懂装懂要切替代评估"一脉相承——对 AI 产出的关键点追问原理，本质就是一次"归属校验 / 替代评估"，能当场把"AI 替他说的"和"他真懂的"区分开。

## 五维打标 → 综合分 映射（参考）

| 五维情况 | 参考综合分 |
|---------|-----------|
| 3+ 个 `+`，其余 `=`，无 `-` | 3.5 ~ 4 |
| 1~2 个 `+`，其余 `=`，无 `-` | 3+ ~ 3.5 |
| 全 `=`，无 `-`，无 `+` | 3（此时走严格原则，**大概率打 3 不通过**） |
| 有 1 个 `-`（非技术深度/技术支撑） | 3 或更低 |
| 技术深度 `-` 或 技术支撑 `-` | 2.5 或更低 |
| 软素质价值观红线 `-` | ≤ 2.5 |
| 多个 `-` | 2 |

注：这是参考映射，不是硬性公式。最终分数必须结合证据强度判断，但**不得违反"严格原则"硬规则**。

> **关于 3 档的裁定**：当五维指向 3+（如 1 个 `+` 其余 `=`）但存在隐性短板（如 AI 应用能力弱、方案选型缺失、关键数据掌控差）把综合压在 3 时，可落 **3**，并由面试官裁定是否通过——若裁定通过，在概览写明"打 3，面试官裁定通过"及理由（如核心维度有真实底子、短板可入职后习得）。**`+` 维度没有自动把综合顶到 3+ 时，要在概览说明是哪些隐性短板拉低了它。**

## 执行流程

1. **定位目标轮次**：
   - 用 `Glob` 扫 `<人名>/[0-9][0-9][0-9]/` 列出所有轮次子目录
   - 含 `asr.md` 的轮次是**目标轮候选**
   - 按前述"判定本次要评估哪一轮"规则确定目标轮。必要时 `AskUserQuestion`。
2. **读取目标轮材料**：
   - `Read` 目标轮的 `asr.md`
   - `Read` 目标轮的 `resume.png`（视觉）；若目标轮无 resume 则查找其他轮的 `resume.png` 复用
   - **只读这两类材料。候选人目录下若有其他面试官的评价文件，一律不读、不纳入。**
3. **岗位定位**：从简历 + asr.md 判断 backend / algorithm / bigdata（或混合）。用户已指定则以用户为准。混合岗位取**严格**的那个标准。
4. **五维打标**：
   - 只基于目标轮的 `asr.md` + `resume.png` 做判断
   - 五个维度分别打 `+`/`=`/`-`，未考核留空
   - 每个标注必须有来自 `asr.md` 的原话证据
5. **综合评分**：根据五维打标 + 证据强度，在 6 档中选一个分数
6. **严格原则校验**：两档之间犹豫 → 取低档，明确在概览里记录"在 X 和 Y 之间犹豫，按严格原则打 X"
7. **结论判定 + 长短处速判**：≥ 3+ 通过；≤ 2.5 不通过；**= 3 时默认倾向不通过，但面试官可基于实际体验显式裁定通过（裁定通过须在概览写明"打 3，面试官裁定通过"+ 理由）**。并在概览用一两句话**点出候选人最大长处 + 最大短处**（各带证据）
8. **面试官复盘评价**：
    - 从 `asr.md` 中提取面试官（用户）的所有发言
    - 按面试官五维评估标准逐项打标（✓/△/✗）
    - 分析时间分配（从时间戳推算）
    - 识别做得好的行为和做得不好的行为（必须有原话证据）
    - 重点看**是否在有限时间内高效判断出候选人的长处与短处**，而非是否面面俱到覆盖每个项目
    - 生成最多 3 条按优先级排列的改进建议
    - 综合给出 A/B/C/D 评分
9. **写报告**：用 `Write` 分别输出两个文件到 **`<人名>/<目标轮目录>/`**：
    - `evaluation.md`：候选人评价（五维打标 + 评分 + 面试记录）
    - `interviewer-review.md`：面试官复盘（五维评估 + 做得好/不好 + 改进建议 + 时间分配）
10. **对话简报**：告知候选人综合分、通过/不通过、核心理由（长处/短处速判）；另附面试官评分和核心改进建议；最终给出两个文件路径

## 评价原则（必须全部遵守）

1. **严格标准**：相邻档之间犹豫一律按低档打；3 默认倾向不通过。唯一例外：3 档可由面试官基于实际面试体验显式裁定通过（须写明理由），其余档位不放水。
2. **证据驱动**：每一个 `+`/`=`/`-` 和每一条优点/风险都**必须引用 `asr.md` 原话或简历原文**（引用原话，不意译）作为依据
3. **主动核对**：把简历上写的项目/技术栈 vs 面试回答交叉比对，重点暴露简历夸大、贡献边界不清
4. **不做政治正确**：不因学校、公司、态度好而放宽判断；也不因背景弱而扣分
5. **区分"没答出"与"答错了"**：没答出可接受（除非是岗位核心基础）；答错了严重扣分
6. **区分"背答案"与"真懂"**：被追问时是否崩溃来判断
7. **不脑补**：简历没写、转写没讲的内容，在报告里明确写"未覆盖"
8. **只看本轮**：评价完全基于本轮 `asr.md` + 简历，不读取、不引用、不被其他面试官的评价影响

## 面试官评价体系

除了评估候选人，本 skill 同时对面试官（即用户本人）的表现进行复盘评价，帮助面试官持续改进面试技巧。

### 面试官评分档位

面试官评分**只能**取以下 4 个值之一：

```
A   B   C   D
```

| 评分 | 含义 |
|------|------|
| A | 优秀：问题路径清晰、节奏好、信息密度高，几乎无改进点 |
| B | 良好：整体合格，有 1-2 个可改进点但不影响评估质量 |
| C | 及格：有明显可改进点，部分维度信息获取不充分 |
| D | 待提高：多个维度有问题，可能影响了对候选人的公正评估 |

### 面试官五维评估

五个维度：**提问路径、节奏控制、追问精准度、面试官纪律、表达效率**

每个维度用 `✓` / `△` / `✗` 标注：

| 标注 | 含义 |
|------|------|
| `✓` | 做得好：该维度表现符合或超出预期 |
| `△` | 需改进：有明确的改进空间，但不严重 |
| `✗` | 有问题：该维度存在明显问题，可能影响面试效果 |

### 各维度含义与判断标准

#### 1. 提问路径

核心原则：**实践结果 → 细节追问 → 原理深挖**，而非概念入手。

- `✓`：问题从候选人的实际工作出发，数字/结果先行，原理从实践中自然引出
- `△`：大部分问题路径正确，但偶尔出现概念入口（如"你了解过X吗"）
- `✗`：频繁以概念/原理开场，候选人被迫先背书再说实践

**判断方法**：统计 `asr.md` 中面试官的问题，计算"从实践切入"vs"从概念切入"的比例。概念入口占比 >30% → `△`，>50% → `✗`。

**概念入口的典型特征**：
- "你了解过 X 吗？"
- "X 的原理是什么？"
- "你能说一下 X 和 Y 的区别吗？"（未关联到具体使用场景）

**实践入口的典型特征**：
- "你们实际遇到 X 了吗？怎么处理的？"
- "这个数字是多少？口径是什么？"
- "上线后有没有出过问题？"

#### 2. 节奏控制

核心原则：**在有限时间内高效判断出候选人的长处与短处**，时间花在能产生判断结论的地方。不要求面面俱到覆盖每个项目，但不能因节奏失控导致关键能力没摸到。

- `✓`：节奏服务于判断——能快速锁定候选人强弱项，编码/核心能力得到验证，没在已问清的点上空耗
- `△`：节奏略有失衡（如某项目深挖过久、或反复纠缠已问清的点），但最终长短处判断仍站得住
- `✗`：节奏失控导致**关键能力未能判断**（如编码没时间落地、核心技术维度完全没摸到、全程困在一个点出不来）

**判断方法**：从 `asr.md` 的时间戳计算各环节用时。下面是**参考节奏**，不是硬指标——只要高效得出了候选人长短处判断，偏离也算 `✓`：
- 自我介绍：3-5 min
- 项目深挖：问出该项目能体现的强弱项即可，问清就切，**不必每个项目都覆盖**
- 编码题：15-25 min（编码是独立能力，尽量留足时间到代码落地）
- Q&A：3-5 min

判定红线：**编码没时间落地、或核心技术维度完全没摸到 → `△` 或 `✗`**（这类缺口直接影响能否判断候选人）。仅仅"简历某个项目没问到"但长短处已判断清楚 → 不扣分。

#### 3. 追问精准度

核心原则：**听到模糊词立刻追具体数字；追问有目的性，不连续轰炸**。

- `✓`：候选人说"大概/可能/差不多"时有追问具体数字；追问有方向性，2-3 个问题后换方向
- `△`：部分模糊回答未被追问，或偶尔出现无目的的连续追问
- `✗`：大量模糊回答被放过；或频繁连续追问让候选人无喘息空间

**需追问的模糊信号词**：
- "大概"、"差不多"、"可能"、"好像"、"左右" → 追"具体多少？"
- "参考了 X" → 追"X 有什么优缺点？你改了哪里？"
- "我们用了 Y" → 追"为什么选 Y 不选 Z？"

#### 4. 面试官纪律

核心原则：**不讨论方案、不补上下文、不下判断、不给偏向性反馈**。

- `✓`：全程保持中立，只问不答（技术层面），Q&A 环节简洁不过度推销
- `△`：偶尔补充了技术上下文或对回答做了轻微评价，但未严重影响
- `✗`：与候选人讨论技术方案；给了明显的偏向性反馈；Q&A 环节过度推销公司/岗位

**红旗行为**（出现任何一条直接 `✗`）：
- 告诉候选人"你应该用 X 方案"
- 对候选人的回答说"不对，应该是..."
- Q&A 环节主动展开超过 3 分钟介绍公司详情
- 结尾暗示通过/不通过结论

#### 5. 表达效率

核心原则：**问一次就够，不解释不重复，善用沉默**。

- `✓`：问题简洁一句话，不重复解释，候选人不懂时由候选人主动追问
- `△`：偶尔重复解释问题（2-3 次），但不频繁
- `✗`：频繁重复/换角度解释同一个问题（>3 次出现），剥夺了考察候选人理解力的机会

**判断方法**：在 `asr.md` 中搜索面试官对同一个问题的多次表述。出现"我再解释一下"、"就是说..."、"换个角度问..." → 计为一次重复。

### 面试官评价的数据来源

面试官评价**完全基于 `asr.md` 中面试官的发言**（面试官的发言通过说话人标识区分）。具体分析：

1. **面试官的问题列表**：提取所有面试官发言中的提问，分析提问路径
2. **时间分配**：通过时间戳计算各环节用时
3. **追问模式**：分析候选人模糊回答后面试官是否跟进
4. **纪律表现**：检查面试官是否有讨论/评价/推销行为
5. **表达模式**：检查面试官是否有重复解释、补充上下文的行为

### 面试官评价与候选人评价的关系

- 面试官评价**不影响**候选人评分。候选人该打多少分就打多少分。
- 面试官评价是**独立的自我复盘**，目的是帮助面试官持续改进。
- 如果面试官的问题导致某维度信息获取不充分（如"技术广度"留空），需在面试官评价中指出——"节奏控制导致未覆盖 X 维度"。

## 候选人评价模板（evaluation.md）

**严格按**下面的结构输出。所有斜体占位符替换为真实内容。不要增加、删除、合并、重命名任何一级/二级/三级标题。五维顺序必须是：业务理解 → 技术支撑 → 技术广度 → 技术深度 → 软素质。

````markdown
# 面试评价报告

## 信息汇总

本轮 *N* 面。

**定性判断：**

维度：业务理解、技术支撑、技术广度、技术深度、软素质。
指标：+ 代表加分项、= 代表持平、- 代表减分项，未考核不记录。

| 维度 | 标注 |
|------|------|
| 业务理解 | *+ / = / - 或留空* |
| 技术支撑 | *+ / = / - 或留空* |
| 技术广度 | *+ / = / - 或留空* |
| 技术深度 | *+ / = / - 或留空* |
| 软素质 | *+ / = / - 或留空* |
````

## 使用示例

用户：
> 评估 `~/interviews/zhang-san`，我做的是 2 面

Claude：
1. `Glob` 扫 `~/interviews/zhang-san/[0-9][0-9][0-9]/`，找到 `001/`、`002/`、`003/`
2. 定位目标轮：`002/asr.md` 存在 → 目标轮为 002（其他轮即使有别的面试官评价文件也忽略）
3. `Read` `002/asr.md` + `002/resume.png`（视觉）
4. 五维打标：业务理解 `=` / 技术支撑 `=` / 技术广度 `=` / 技术深度 `-` / 软素质 `=`
5. 技术深度 `-` 的证据：简历写"精通 MySQL"，被问"聚簇索引回表"卡壳（引用 `asr.md` 原话）
6. 综合分在 3 和 3+ 之间犹豫 → 严格原则 → 打 **3**；速判：长处=系统设计链路完整，短处=MySQL 基础不扎实
7. 面试官复盘：提问路径 `✓`，节奏控制 `△`（项目1 深挖偏久，但长短处已判断清楚），追问精准度 `✓`，面试官纪律 `✓`，表达效率 `△`（数据链路问题解释了 3 遍）。综合 **B**。
8. `Write` 两个文件到 `~/interviews/zhang-san/002/`：
    - `evaluation.md`（候选人评价）
    - `interviewer-review.md`（面试官复盘）
9. 对话简报：
    - 候选人：**3，非 3+，不通过**。核心理由：MySQL 基础不扎实。
    - 面试官：**B**。核心改进：编码环节留足时间到代码落地。

### 示例 2：用户直接贴飞书招聘链接（端到端）

用户：
> 参考上一个候选人的处理方式，处理这个候选人 https://xxx.feishu.cn/hire/talent/765…?application_id=765…

Claude：
1. **阶段 0 采集**：Chrome 打开链接 → title 取候选人姓名 → 拼音建目录 `~/github/interviews/<拼音>/001/`
2. 页面内 fetch `get_default_resume` API → 下载 PDF → `scripts/pdf2png.swift` 渲染 `resume.png` → 缩略图目检
3. 点击「面试速记」→ 滚动到底验证全量 → 抽取全部消息 → 校验（条数/首末/说话人）→ 生成 `asr.md`
4. **并行**发起两个子代理：候选人五维证据提取 + 面试官复盘证据分析；主对话同时亲读 asr.md + 简历
5. 确认目标岗位对标职级（如 2-1）→ 按职级校准打分 → 写 `evaluation.md` + `interviewer-review.md`
6. 对话简报（评分/通过与否/长短处/面试官评分/改进建议/文件路径）
7. 若用户基于现场观察裁定改分 → 更新报告并在概览留痕初判与裁定理由

## 不做什么

- ❌ 不联网搜索候选人信息（隐私 & 偏见）
- ❌ 不对背景（学校/公司/籍贯/性别）做倾向性评价
- ❌ 不脑补简历没写、转写没讲的信息
- ❌ 不为凑字数编造"优点"——没有就写"无突出亮点"
- ❌ 不使用 2/2.5/3/3+/3.5/4 以外的分数
- ❌ 不在 borderline 放水打高档
- ❌ 不给"未考核"维度勉强打 `=`（直接留空）
- ❌ 不读取、不引用其他面试官的评价文件——每轮只基于本轮 `asr.md` + 简历独立成文
- ❌ 不强求面面俱到覆盖每个项目——优先在有限时间内高效判断出候选人长短处
- ❌ 不把报告写到错误的路径——候选人评价必须是 `<人名>/<目标轮目录>/evaluation.md`，面试官复盘必须是 `<人名>/<目标轮目录>/interviewer-review.md`
