# Interview Comment（面试评价生成）

按团队固定模板生成后端/算法/大数据研发岗位的面试评价。Claude 亲自读简历截图（视觉）+ 语音转写 + 其他面试官评价文本，五维打标 + 综合评分 + 跨轮交叉参考 + 按模板输出 Markdown 报告。

## 触发

用户说类似：

- "评估 `<人名目录>` 的第 N 轮"
- "帮我写 `<人名>/002` 的面试评价"
- "运行 interview-comment，候选人在 `<人名目录>`，我做的是 2 面"

## 输入目录结构（固定）

顶层是候选人标识（姓名 / 缩写 / 工号），下面每一轮面试一个子目录，命名为 `001`、`002`、`003`、`004`，分别代表 1~4 面。

```
<人名>/
├── 001/
│   ├── resume.png        # 简历截图（可能存在）
│   ├── asr.md            # 该轮面试语音转文字（若用户本人是该轮面试官）
│   └── review-res.md     # 该轮面试官的评价文本（若是其他面试官）
├── 002/
│   ├── resume.png
│   ├── asr.md            # ← 假设用户本次做的是 2 面
│   └── ...
├── 003/
│   └── review-res.md     # 其他面试官已完成的后轮评价（若存在）
└── ...
```

**文件命名规则（严格匹配，不要用模糊搜索）**：

| 文件 | 含义 | 出现在 |
|------|------|--------|
| `resume.png` | 简历截图 | 通常每轮都有（允许多轮共享同一张） |
| `asr.md` | 该轮面试的语音转文字 | **仅用户本人担任面试官的轮次** |
| `review-res.md` | 该轮面试官的评价文本 | **仅其他面试官担任面试官的轮次** |

### 判定"本次要评估哪一轮"

- 只有含 `asr.md` 的轮次才是**本次评估的目标轮**（评价要写到那一轮的目录里）
- 含 `review-res.md` 但无 `asr.md` 的轮次是**参考轮**（只读取，用于交叉参考）
- 如果多个轮次都有 `asr.md` → 用 `AskUserQuestion` 让用户指定要评估哪一轮
- 如果没有任何一轮有 `asr.md` → 报错退出（缺少必要材料）
- 用户已在对话里明确指定轮次（如"帮我写 002"）→ 以用户指定为准

### 必需文件与可选文件

- **必需**（目标轮必有）：`asr.md`
- **强烈建议**（目标轮应有）：`resume.png`。若目标轮没有，查找其他轮是否有 `resume.png`，有就共用；全都没有则报错退出。
- **可选**（参考轮）：其他轮次的 `review-res.md`，有多少读多少

缺少 `asr.md` 或所有轮都无 `resume.png` → 直接报错退出，不脑补。

## 评分体系（核心）

### 分数档位

候选人的综合评分**只能**取以下 7 个值之一：

```
2.5   2.75   3   3.25   3.5   3.75   4
```

**禁止**输出 3.1、3.4、3.6 等非档位分数。

### 档位含义

| 分数 | 含义 |
|------|------|
| 4 | 顶尖候选人，各维度均有明显亮点，能立即独当一面 |
| 3.75 | 非常优秀，2+ 个维度有显著加分项 |
| 3.5 | 优秀，有明确亮点，轻微待提升 |
| 3.25 | 合格偏上，能做事，但亮点不突出（3+ 通过线） |
| 3 | 合格线边缘，能过 borderline 但策略上不通过 |
| 2.75 | 有明显硬伤，不通过 |
| 2.5 | 明显不符合岗位要求 |

### 3+ 的定义

**3+ = 3.25 及以上**（包含 3.25、3.5、3.75、4）

**通过标准**：综合评分 ≥ 3.25 即通过。

### 严格打分原则（硬规则）

1. borderline 一律按**低档**打：
   - 在 **3 和 3.25 之间犹豫 → 打 3（不通过）**
   - 在 3.25 和 3.5 之间犹豫 → 打 3.25
   - 在 3.5 和 3.75 之间犹豫 → 打 3.5
   - 任何"可打可不打高一档"的情况 → 打低一档
2. 宁可错杀，不可错放
3. 给 3.25 必须说得出**至少 2 条**证据确凿的加分理由
4. 给 ≥3.5 必须说得出**至少 1 条**显著亮点（加分项，非持平）

## 五维度定性判断

五个维度：**业务理解、技术支撑、技术广度、技术深度、软素质**

每个维度用 `+` / `=` / `-` 标注：

| 标注 | 含义 |
|------|------|
| `+` | 加分项：该维度有明显亮点，有证据支撑 |
| `=` | 持平：符合岗位预期，无明显亮点或硬伤 |
| `-` | 减分项：该维度有明显不足或答错 |
| 未考核 | 本轮未覆盖 → **留空、不记录**（不要勉强给 `=`） |

### 各维度含义

#### 1. 业务理解

候选人对所做业务的理解深度。

- 核心判断：能否讲清业务目标 → 技术方案的映射
- 是否用业务指标而非单纯技术指标衡量成败
- 是否关心 ROI、成本、SLA
- **后端**：能否讲清接口/系统服务的上游业务、量级、关键 SLO
- **算法**：能否讲清模型优化指标 vs 业务指标的差异、badcase 驱动迭代经验
- **大数据**：能否讲清数据产品的消费方、SLA、数据 → 决策链路

#### 2. 技术支撑

候选人的技术能力能否真正支撑业务目标。

- 核心判断：技术选型的决策链路是否清楚、trade-off 意识、上线后真实表现
- 是否能讲出"为什么不选另一种方案"
- 是否有量级估算习惯
- `-` 红旗：把"我们用 Redis"当成设计题答案；扩展性追问直接卡死

#### 3. 技术广度

候选人技术栈的覆盖面。

- 核心判断：岗位相邻领域是否接触过、能否把不同组件串起来
- **注意**：广度 ≠ 列一堆名词，能讲清每个东西的**适用场景**才算广度
- `-` 红旗：简历列一堆"熟悉 X/Y/Z"，但任意挑一个都说不出和同类的区别

#### 4. 技术深度

候选人在至少一个领域的深度。

- 核心判断：被追问原理时能答到第几层
- 简历上写"精通 / 熟悉 X"是否经得起追问
- 关键区分：是**背答案**还是**真懂**（追问时是否崩溃/绕开/换话题）
- 区分"没答出"（知识盲区，可接受）vs"答错了"（原理讲反，严重扣分）
- `-` 红旗：把核心概念讲反；简历明确写"精通 X"被问 X 基础原理答不出

**岗位细化关注点**：

- **后端**：并发与锁、MySQL 索引/事务/MVCC、缓存一致性、MQ 语义、分布式 CAP
- **算法**：模型原理（loss/优化器/反向传播）、训练/推理工程瓶颈、实验设计与指标体系
- **大数据**：Spark/Flink 执行原理、shuffle、数据倾斜、state/exactly-once、数据建模分层

#### 5. 软素质

沟通、学习、价值观、稳定性的综合判断。

- 回答是否结构化（先结论再展开）
- 对追问的反应（积极重组 vs 不耐烦/抵触）
- 跳槽节奏是否合理、对前司是否明显情绪化
- 对"近期学了什么"能否答出有说服力内容
- **红线**：价值观问题（如对同事的不当评价、歧视言论、"卷同事"当荣誉）→ 直接 `-`，且综合分不得高于 2.75

## 五维打标 → 综合分 映射（参考）

| 五维情况 | 参考综合分 |
|---------|-----------|
| 3+ 个 `+`，其余 `=`，无 `-` | 3.75 ~ 4 |
| 1~2 个 `+`，其余 `=`，无 `-` | 3.5 |
| 全 `=`，无 `-`，无 `+` | 3 ~ 3.25（此时走严格原则，**大概率打 3 不通过**） |
| 有 1 个 `-`（非技术深度/技术支撑） | 3 或更低 |
| 技术深度 `-` 或 技术支撑 `-` | 2.75 或更低 |
| 软素质价值观红线 `-` | ≤ 2.75 |
| 多个 `-` | 2.5 |

注：这是参考映射，不是硬性公式。最终分数必须结合证据强度判断，但**不得违反"严格原则"硬规则**。

## 执行流程

1. **定位目标轮次**：
   - 用 `Glob` 扫 `<人名>/[0-9][0-9][0-9]/` 列出所有轮次子目录
   - 分类：含 `asr.md` 的是**目标轮候选**，含 `review-res.md` 的是**参考轮**
   - 按前述"判定本次要评估哪一轮"规则确定目标轮。必要时 `AskUserQuestion`。
2. **读取目标轮材料**：
   - `Read` 目标轮的 `asr.md`
   - `Read` 目标轮的 `resume.png`（视觉）；若目标轮无 resume 则查找其他轮的 `resume.png` 复用
3. **读取参考轮材料**（跨轮交叉参考）：
   - 对每个参考轮，`Read` 其 `review-res.md`，解析出其他面试官给出的：
     - 分数与通过/不通过结论
     - 五维标注（如果该轮用了同一套体系）
     - 优点 / 待提升点 / 风险点 / 后续关注
   - 如果参考轮评价用的不是本 skill 的五维体系，做语义映射，不强行套
4. **岗位定位**：从简历 + asr.md 判断 backend / algorithm / bigdata（或混合）。用户已指定则以用户为准。混合岗位取**严格**的那个标准。
5. **独立五维打标**（先不看参考轮）：
   - 只基于目标轮的 `asr.md` + `resume.png` 做独立判断
   - 五个维度分别打 `+`/`=`/`-`，未考核留空
   - 每个标注必须有来自 `asr.md` 的原话证据
6. **跨轮交叉校对**（此时才引入参考轮观点）：
   - 把独立判断 vs 参考轮评价做对比
   - 走"其他面试官评价参考处理原则"（见下一节）
   - 必要时调整自己的标注 / 风险点 / 后续关注
7. **综合评分**：根据校对后的五维打标 + 证据强度，在 7 档中选一个分数
8. **严格原则校验**：两档之间犹豫 → 取低档，明确在概览里记录"在 X 和 Y 之间犹豫，按严格原则打 X"
9. **结论判定**：≥3.25 通过，<3.25 不通过
10. **写报告**：用 `Write` 输出到 **`<人名>/<目标轮目录>/evaluation.md`**（如 `zhang-san/002/evaluation.md`）
11. **对话简报**：告知综合分、通过/不通过、核心理由、与参考轮的一致/分歧、文件路径

## 其他面试官评价参考处理原则

`review-res.md` 是**参考**，不是 ground truth。使用它的原则：

1. **先独立判断，再引入参考**：
   - 必须**先**基于 `asr.md` 完成独立五维打标，**之后**才读参考轮评价做校对
   - 防止被其他面试官的结论先入为主带偏

2. **三种情况的处理方式**：

   | 独立判断 vs 参考评价 | 处理方式 |
   |----------------------|---------|
   | **一致**（结论相同，理由重合） | 在报告"跨轮交叉参考"节说明一致；信心提升，维持独立判断 |
   | **参考发现了我没发现的点**（补充） | 回到 `asr.md` 核验：若本轮有证据 → 补入本轮报告；若本轮无证据 → 写入"后续面试重点关注"让下轮深挖 |
   | **参考与我分歧**（校正） | 回到 `asr.md` 原文核对证据，按下面的分歧处理规则决策 |

3. **分歧处理规则**（优先级从高到低）：
   1. **证据驱动**：两边谁的证据更强、更能引用原话？选证据强的一方。
   2. **严格优先**：如果两边证据都站得住，选**更严格**的那一方（与本 skill 严格原则对齐）。例如参考轮打 3.5，我判 3，证据势均力敌 → 维持 3。
   3. **岗位边界**：其他面试官可能考察的是不同维度（1 面算法基础 vs 3 面系统设计），不要把他们对 X 维度的判断直接套到本轮的 Y 维度。
   4. **时间先后不等于权威**：不要因为参考轮是后一轮就更信它；也不要因为是前一轮就更信。只看证据。

4. **必须在报告中显式标注**：
   - 任何来自参考轮的信息（补充事实、风险、后续关注）都要注明来源轮次，如"（1 面面试官指出）"
   - 对参考轮的分歧要在"跨轮交叉参考"节明说，并给出维持己见或修正的理由

5. **不做的事**：
   - ❌ 不盲从其他面试官的分数 / 结论
   - ❌ 不因其他面试官打了高分就把自己判的 `-` 改成 `=`（除非有 `asr.md` 原文证据）
   - ❌ 不把参考轮里候选人对该轮面试官的表现直接当成本轮的证据（不同轮不同表现）
   - ❌ 不复制粘贴其他面试官的用词——用自己的话重述

## 评价原则（必须全部遵守）

1. **严格标准**：borderline 一律按不通过处理
2. **证据驱动**：每一个 `+`/`=`/`-` 和每一条优点/风险都**必须引用 `asr.md` 原话或简历原文**（引用原话，不意译）作为依据。跨轮补充的内容需注明参考轮来源。
3. **主动核对**：把简历上写的项目/技术栈 vs 面试回答交叉比对，重点暴露简历夸大、贡献边界不清
4. **不做政治正确**：不因学校、公司、态度好而放宽判断；也不因背景弱而扣分
5. **区分"没答出"与"答错了"**：没答出可接受（除非是岗位核心基础）；答错了严重扣分
6. **区分"背答案"与"真懂"**：被追问时是否崩溃来判断
7. **不脑补**：简历没写、转写没讲的内容，在报告里明确写"未覆盖"
8. **先独立再参考**：永远先基于本轮 `asr.md` 独立打标，再引入其他面试官的评价做校对

## 不做什么

- ❌ 不联网搜索候选人信息（隐私 & 偏见）
- ❌ 不对背景（学校/公司/籍贯/性别）做倾向性评价
- ❌ 不脑补简历没写、转写没讲的信息
- ❌ 不为凑字数编造"优点"——没有就写"无突出亮点"
- ❌ 不使用 2.5/2.75/3/3.25/3.5/3.75/4 以外的分数
- ❌ 不在 borderline 放水打高档
- ❌ 不给"未考核"维度勉强打 `=`（直接留空）
- ❌ 不盲从其他面试官的结论；不把参考轮内容当成本轮的证据
- ❌ 不先读参考轮评价再做独立打标（会被先入为主带偏）
- ❌ 不把报告写到错误的路径——必须是 `<人名>/<目标轮目录>/evaluation.md`
